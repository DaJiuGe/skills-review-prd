# SaaS 权限管理系统 PRD

## 1. 项目背景与目标

本系统为多租户 SaaS 平台提供统一的权限管理能力，支持租户在组织内定义角色、分配权限、管理成员，并集成企业身份提供商（IdP）实现单点登录与审计合规。

## 2. 目标用户与角色

- **租户管理员（TenantAdmin）**：创建角色、为组织成员分配角色、查看权限报表。
- **组织普通成员（Member）**：根据被授予的权限使用平台功能。
- **安全审计员（Auditor）**：查看角色变更日志、审计授权链路。
- **IdP 同步服务（IdPSync）**：从企业身份提供商同步用户与组织关系。

## 3. 核心概念模型

系统核心概念包括：

- **Role（角色）**：一组权限的集合，用于描述用户在系统中的职能身份。PRD 部分章节也使用“角色定义”一词。
- **Permission（权限）**：对资源执行某种操作的许可，例如 `order:read`、`order:delete`。部分章节简称其为“权限项”。
- **Resource（资源）**：权限作用的对象，例如订单、商品、报表。部分章节为了行文简洁，将 Resource 也称作“权限”。
- **User（用户）**：平台账号持有者，可被加入多个 Organization。
- **Organization（组织）**：租户下的组织单元，包含成员、角色与资源的关联关系。

“角色 / Role / 权限 / Permission / 资源 / Resource”在多处交叉混用，术语边界需要澄清。

## 4. 角色继承与授权流程

角色支持继承关系，子角色自动拥有父角色的所有权限。租户管理员执行 **CreateRole** 命令创建角色，执行 **AssignRoleToUser** 命令将角色授予组织成员。

角色继承可能出现循环：例如“部门管理员”继承“普通成员”，而“普通成员”在跨部门协作场景下又被配置为继承“部门管理员”。系统在鉴权时需要递归解析角色链，循环继承将导致授权计算无限递归。

鉴权流程：用户访问资源时，系统根据用户所属角色展开权限，检查是否具备对应 Resource 的 Permission。

## 5. 组织成员关系

用户与组织的关系由以下行为维护：

- **AddUserToOrganization** 命令将用户加入组织。
- **RemoveUserFromOrganization** 命令将用户从组织移除。
- 用户被加入组织时触发 **UserJoinedOrganization** 事件；被移除时触发 **UserLeftOrganization** 事件。

User 聚合维护用户账号、状态、联系方式等属性；Organization 聚合维护组织基本信息与成员列表。两个聚合都掌握“用户是否属于组织”这一事实，成员关系的唯一权威来源未明确。

管理员可以撤销用户的角色，但 PRD 没有定义对应的 **RevokeRole** 命令或事件。

## 6. 外部系统集成

- **IdentityProvider（身份提供商，IdP）**：通过 SAML / OIDC 协议实现企业用户同步与单点登录。
- **AuditLogService（审计日志服务）**：记录角色创建、分配、继承变更、权限使用日志，用于合规审计。

## 7. 读模型与查询

- **UserPermissionView（用户权限视图）**：展示指定用户在指定组织下拥有的全部展开权限，用于权限预览与问题排查。
- **OrganizationRoleTree（组织角色树）**：展示组织内角色继承关系、每个角色下的成员数量、继承来源。
- **PermissionAssignmentView（权限分配视图）**：展示某条 Permission 被哪些 Role 持有、哪些 User 最终生效。

## 8. 审计与合规

所有角色与权限变更必须记录审计日志，包括操作人、变更前快照、变更后快照。审计日志通过 **AuditLogService** 异步写入。

撤销角色、删除角色等敏感操作需要二次确认，但 PRD 未给出具体交互细节与命令定义。
